Introduction

When it comes to programming, I've found myself studying hard as of late, and doing nothing but. In fact, I think it's been at least a couple of weeks (probably longer) since I last worked on an actual project, which is the part of programming that I'm passionate about. Being able to create something from virtually nothing is incredibly satisfying and fulfilling for me (not to mention fun as hell).

I ended up wanting to take a little bit of a break, during which I wanted to build something new. Because this time spent away from studying was meant to be a break, I decided to go for a project that was fun and not too difficult. After all, practice makes perfect, no matter how "easy" something might be.

I found this partiular project idea on Rosetta Code. It's called "21 Game", and it's an extremely simple game. Two players (in this case, the player vs. the computer) take turns playing a number. They can choose to play either 1, 2, or 3. As they alternate turns, their numbers are added to the game's running total. The game ends when this running total reaches 21, with the player who made it 21 is declared the winner.

The original instructions for this project, which, again, is from Rosetta Code, are as follows:

21 Game Instructions

  1. 21 is a two player game, the game is played by choosing a number (1, 2, or 3) to be added to the running total.
  2. The game is won by the player whose chosen number causes the running total to reach exactly 21.
  3. The running total starts at zero. One player will be the computer.
  4. Players alternate supplying a number to be added to the running total.
Task

Write a computer program that will:

  • do the prompting (or provide a button menu)
  • check for errors and display appropriate error messages
  • do the additions (add a chosen number to the running total)
  • display the running total
  • provide a mechanism for the player to quit/exit/halt/stop/close the program
  • issue a notification when there is a winner
  • determine who goes first (maybe a random or user choice, or can be specified when the game begins).

Making Mistakes

Because this project is pretty simple, I didn't feel the need to do a lot of planning. I did quickly map out the logic I thought I would use to code the mechanics of the game, which ended up being incorrect. When I ran the code, the player's and computer's first turns worked properly, but after that things went all wonky. I do understand why it happened now, which I'll discuss in the next section.

My original program went something like this (although note that this is in a very broad and general sense):

  • Player clicks 'Start Game' button to get things going.
  • Clicking the button triggers a modal to pop up with 3 buttons, which are for the player to use to deicde who goes first (or to make the program decide at random). Clicking any of these buttons closes the modal window, and the game is free to commence.

My first error in logic was trying to implement three functions that looked like this:

function playerStartFunc() {
    playerFirst = true;
    computerFirst = false;
    randomFirst = false;
}

function computerStartFunc() {
    playerFirst = true;
    computerFirst = false;
    randomFirst = false;
}

function randomStartFunc() {
    playerFirst = true;
    computerFirst = false;
    randomFirst = false;
}

In case you couldn't tell, my initial (and very flawed) thinking was that the 3 buttons would each trigger a different function, which was how I would get the program to have different "start conditions". The code inside of each of these functions are just 3 variables getting set to true or false, depending on who is supposed to start the game. I mean, it's too much for something so simple. I was tired! I normally do a little better, I swear.

These functions were then followed by an additional function called choosingWhoGoesFirst():

Don't even ask me what the hell I was thinking with this function, because I don't know. It's a complete mess to try and make sense of. Let's just...move on.

As much of a mess as it is, this isn't even the main problem with my original code. That came from three additional functions called playerTurn(), computerTurn(), and continueGame(). I also had variables that I called computerWent and playerWent (which, despite how their names sound, are not functions about using the bathroom).   1. The player would click the Submit button, which, when clicked, would trigger playerTurn(), which:
    1. Linked the input value to a variable called playerNumber.
    2. Added playerNumber to the running total.
    3. Created the player's turn announcement and appended it to the correct element on the page.
    4. Updated the running total to reflect the player's play.
    At the end of this function, another function, continueGame(), was called.
  2. continueGame() checked to see if the running total was 21.
    • If not: If computerWent was set to true, playerTurn() was called, and vice versa; if playerWent was set to true, then computerTurn() was called instead. (I don't know why, but I thought this was needed in order to continue the game along how I wanted it to.)
    • If so: If playerWent was set to true, this meant the player had turned the number into 21, and was thus the winner. The same went for computerWent, of course: if it was set to true, then the computer was the winner. The function would then put the "winner announcement" on the screen.
  3. computerTurn() was nearly identical to playerTurn(), except its number was, of course, randomly chosen instead of taken from the input value.

Do you see the problem with this yet? It's okay if not. Maybe you will after I tell you how my program chose to act with this version of the code.

So the first turn would work as intended. The player and computer both had a turn, and their numbers were added to the running total.

However, the next time the player tried to submit a number, the game would break.

  • The first additional time: The player's number and the computer's number would both be submitted as normal. But then an extra turn would be taken automatically, with the computer submitting another number, and the player "submitting" NaN. This would make the running total NaN from thereon after.
  • Upon further testing (aka trying to go through a 3rd round), the program would break even further. Once again ,the computer and the player would submit their intedned numbers. However, the program would then force an additional three turns to go through, all with the player's number being NaN. The more I tried to play a turn, the more these additional turns would increase. It was out of control.

When I reflected back on the program today (which, mind you, was after I'd written the official working code), I realized something. I remembered that the player's turn was supposed to be triggered by pressing the Submit button. However, since continueGame() also called playerTurn(), these extra turns would be taken. I don't quite understand why the amount of additional turns increased in number each time, though.

I also realized why the player's number would be submitted as NaN after the first submission. It was because there was no new input value to refer to! After all, the player's number was taken from the input box's value. So when the Submit button was clicked, the player would have already typed a number into the input box, and it would apply. After that, my program clears the input box's value to nothing (essentially "resetting" it), and that's why it would then be NaN.

Anyway, I took a short break after this miniature "fiasco" (before realizing why it was broken, leaving me annoyed and confused) and drafted a couple of new ideas on paper. When I came back, I managed to rewrite the program in a way that, to my relief, worked as intended!

Once I got the program to work as intended, I set to work coding additional things such as keyboard shortcuts, a restartGame() function, and more. This brings us to our first function. This is the only function written purely in jQuery in this script, and it doesn't have a name. That's because I heavily referenced the jQuery UI API Documentation, essentially stitching together various code pieces to pull together this function.

  • show: true - included to ensure that a "fade in" animation is what happens when the modal is called.
  • buttons - Each object in this array represents one of the three buttons shown on the modal - one to pick the player to go first, one to pick the computer to go first, and one to let the program choose at random.
  • modal: true - indicates that this should be a modal (and not just a dialog window)
  • autoOpen: false - set so that the modal won't open immediately upon initialization, but instead waits until the open method is called.
  • open - Opens the dialog.
    • Includes a line that hides the original close button, which is usually located in the titlebar of the modal window. This was done because it forces the player to click one of the buttons, which will then close the modal.
  • dialogClass: "no-close" - included because it, too, is necessary to hide the close button.
  • resizable: false - this means that the modal window is unable to be resized by the user. If this is set to true, the user can grab the edges of the modal window and drag them to resize the window.
  • title: "Select Who Starts" - Sets this as the title of the modal, shown in the titlebar.
  • $("#startgame").click - This function specifies that the modal window opens when the Start Game button is clicked.

Here we have the startGame() function. It includes the starter parameter, which stands for whomever is chosen to start the game. This function is called by the buttons located in the modal window.

This function consists of several if/else conditions. The first four are for changing the display value of various elements (the input box, the Submit button, the Start Game button, and the running total box). In other words: when the player opens the webpage, they see just the Start Game button underneath the instructions. Clicking on the button makes the button disappear, and makes all of the other elements appear.

The final if/else condition is for specifying what happens as a result of who is starting the game. If "random" was chosen, the function chooses a number randomly between 1 and 2. If the number turns out to be 1, then the computer starts the game, and if the number is 2, then the player starts the game. If the computer is chosen to start the game, then the computerTurn() funciton is run. If the player is chosen, well, nothing happens; the game elements are no the page, waiting to be interacted with, and it's the clicking of the Submit button that makes everything happen.

Next we have the playRound() function, which, if you'll recall, is called when the Submit button is clicked.

Starting things off is an if statement. This statement says that if the player's input value is a) less than 1, b) more than 3, or c) anything other than a digit (including letters and special characters), then the program should run an "alert" containing the specified error message.

However, if the player has successfully submitted a valid play, then the following happens:

  • The playerNumber variable is set via the input box's submtited value, which includes parseInt to ensure that it is taken as an integer instead of a string.
  • turns += 1 - The turns variable increases by 1.
  • Two span elements are created - one for the Turns column, the other for the Player Turns column. These span eleemnts contain the current turn and the player's number, respectively, and are appended to their appropriate columns.
  • The player's number is added to the running total, which is then updated.
  • If the running total is now 21, then the winner is the Player, and the endGame() function runs. If not, then computerTurn() runs instead.

Regardless of which part of the if statement runs, the final part of this function ensures that the input box's value is reset (so that it's empty), and then puts focus on that input box so that the player doesn't have to click on it again every turn.

function playRound() {
    let playerNumber = parseInt($("#playerinput").val());
    turns += 1;
    $(".main").addClass('regback').removeClass('winback errorback');
    $(".resultsbox").removeClass("error green");
    $(".submitbtn").removeClass("btnbackerror btnbackgreen");
    $(".restart").removeClass("btnbackerror btnbackgreen");

    if((runningTotal + playerNumber) > 21) {
        $(".main").addClass("errorback").removeClass('regback');
        $(".resultsbox").addClass("error");
        $(".submitbtn").addClass("btnbackerror");
        $(".restart").addClass("btnbackerror");
        $(".turnsbox").hide();
        $(".errormsg").show().html("<strong>ERROR:</strong> The aim is to make the number 21. 
        If you play that number, you'll go over 21. Try again - you're about to win!");
        $(".resultsbox").css("text-align", "center");
    } else {
        if(gameOver === true) {
            $(".main").addClass("errorback").removeClass('regback');
            $(".resultsbox").addClass("error");
            $(".submitbtn").addClass("btnbackerror");
            $(".restart").addClass("btnbackerror");
            $(".turnsbox").hide();
            $(".errormsg").show().html("ERROR: The game is over. You must press the Restart button to play again.");
            $(".resultsbox").css("text-align", "center");
        } else if($("input").val() < 1 || $("input").val() > 3 || $("input").val().match(/\D/gi)) {
            $(".main").addClass("errorback").removeClass('regback');
            $(".resultsbox").addClass("error");
            $(".submitbtn").addClass("btnbackerror");
            $(".restart").addClass("btnbackerror");
            $(".turnsbox").hide();
            $(".errormsg").show().html("ERROR: Invalid entry. Please enter a valid number (1, 2, or 3).");
            $(".resultsbox").css("text-align", "center");
        } else {
            $(".runningtotalbox").show();
            $(".turnsbox").show();
            $(".errormsg").hide();
            $(".resultsbox").removeClass("error");
            $(".restart").removeClass("btnbackerror");
            $(".submitbtn").removeClass("btnbackerror");
            $("<span>" + playerNumber + "</span>").appendTo( $(".player-turns") );
            $("<span>" + turns + "</span>").appendTo( $(".turns") );
            runningTotal += parseInt(playerNumber);
            $(".runningtotal").text(runningTotal);
            
            if(runningTotal === 21) {
                winner = 'Player';
                endGame();
            } else {
                computerTurn();
            }
        }
    }

    $("#playerinput").val(' ');
    $("#playerinput").focus(' ');
}

This brings us to this behemoth of a function, but there truly isn't as much to this function as it seems. A lot of these lines of code are simply there to add and remove various classes from elements so that the different color themes are applied (i.e., if there is an error, or if the game ends).

  • The playerNumber variable is initialized with the player's number, which is taken from the input box's vaule. parseInt is run on this value so that it is taken in as an integer instead of as a string.
  • All possible prior error and win condition classes are removed from their respective elements.
  • We come to an if statement that says if the running number is over 21, it results in the red "Error" theme being applied, plus an error message that reminds the player that they're aiming to make the number 21 and that they can win if they pick the correct number. If the number doesn't exceed 21, we are free to continue.
  • We are met with another if staement, this one containing various conditions.
    • The first condition is for whether gameOver === true. gameOver is a variable that is initially initialized as false; however, when the endGame() function runs, the variable is set to true to signal that the game is over. If it is true, this generates the error theme with an error that tells the player that the game is over, and that they must press Reset to restart the game. This was implemented after I realized that the user could press the Submit button even after the game was supposed to be over, generating unwanted results.
    • Next, we ask if the player's input value is invalid. It's invalid if their number is below 1, above 3, or is something that isn't a number at all. This, too, results in an error with the same error theme, with this error telling the player that they've tried to submit an invalid play, and that they should play a number from 1 - 3 (inclusive).
    • Finally, if none of these errors occur, we make sure that all "error" classes are removed from the appropriate elements. We then create <span> elements, which are appended to both the Turns column and the Player Turns column. The one in the Turns column contains the current turn number, and the one in the Player Turns column contains the player's number that they played during this turn.
    • The player's number is added to the running total, which is updated accordingly.
    • Via another if statement, if the running total is 21, the player is deemed the winner, and the endGame() function runs. If not, the comptuerTurn() function runs instead.
  • The function ends with the input value getting reset (so that it's empty) and the input box being focusd on so that the player doesn't have to click on it again.

Finally, our jQuery ends with these miscellaneous functions, which program the game's keyboard shortcuts and click events for the buttons.

  • If the Enter key is pressed while the input box is focused, the playRound() function runs. This is the same as pressing the Submit button.
  • If the Shift + R key combination is pressed, it's the same as the Reset button being clicked.
  • Shift + S starts the game (and is the same as pressing the Start Game button).
  • When the Reset button is clicked:
    • If the input box is disabled (which only happens when the game is over), the page reloads without a fuss.
    • If the input box is not disabled, the program sends a confirmation message to the player, asking them to confirm that they're sure they want to restart a game, which they're in the middle of. If they confirm, then the page reloads, restarting the game.
